Phase 4 — API Ingestion

Connector-based pipeline for scaling beyond manual uploads

Core idea: connectors feed the same import pipeline

Rather than treating each API as a special case, Phase 4 introduces a connector abstraction. Each connector knows how to fetch data and normalise it into rows — then hands off to the exact same period assignment → column review → field mapping flow already built. The downstream experience is identical to CSV or Drive.

Phase 4a — Google Cloud Console connector

Connects via Google APIs (Analytics, Search Console, or custom project metrics depending on what data is being pulled). Auth via OAuth service account or user token. Connector exposes a query builder — select metric, dimension, date range — and delivers rows to the pipeline.

ConnectorGoogle Cloud Console
AuthOAuth 2.0 (per user)
Fetch triggerManual pull (scheduled pull in future)
OutputRows → ImportBatch (kind: api)

Phase 4b — Profound connector

Connects to Profound's API using an API key stored per user or org. Connector handles pagination, rate limits, and field normalisation. Same manual pull trigger to start — scheduled pulls can be added once the pattern is proven.

ConnectorProfound
AuthAPI key (stored in settings, encrypted at rest)
Fetch triggerManual pull (scheduled pull in future)
OutputRows → ImportBatch (kind: api)

What Phase 4 does NOT include (yet)

  • Scheduled / automatic pulls — manual trigger only in Phase 4
  • Webhook-based push ingestion
  • Arbitrary third-party connector marketplace
  • Real-time streaming data

These are natural Phase 5+ additions once the connector abstraction is proven with two real sources.
